# AmneziaGeo

[English](README.md) · **Русский**

> Лёгкий VPN-клиент на **AmneziaWG** с **гео-осознанным раздельным туннелированием** — заворачивать в DPI-устойчивый туннель только то, что вы выбрали (по домену, категории доменов, стране, CIDR или приложению), а всё остальное пускать напрямую.

AmneziaGeo — десктопный VPN-клиент на **AmneziaWG**, DPI-устойчивом форке WireGuard. В отличие от обычного WireGuard-клиента, он решает, *что именно* идёт в туннель, — **по домену, категории geosite, стране, CIDR или приложению**, держит это решение корректным, пока CDN крутят свои IP, и пробивается сквозь враждебные сети, туннелируясь через WebSocket / TCP 443, когда голый UDP заблокирован.

---

## Какую проблему решает

WireGuard (и AmneziaWG) маршрутизирует исключительно по **IP** — *cryptokey routing* через `AllowedIPs` пира. У протокола **нет понятия домена**. Наивно развернуть категорию `geosite:` в замороженный снимок IP не работает на практике: CDN (Cloudflare и прочие) крутят A-записи и отдают разные IP для разных поддоменов, поэтому приложение попадает на адрес, который никогда не был замаршрутизирован, — и трафик утекает напрямую.

AmneziaGeo решает это **живым перехватом DNS**. Локальный DNS-прокси наблюдает *реальные* ответы, которые получает система, и программирует маршрутизируемый набор туннеля (`AllowedIPs` + маршруты ОС) **динамически, до того как приложение откроет соединение** — это идея sing-box / v2ray «маршрутизировать по реальному DNS», надстроенная над cryptokey routing WireGuard.

**Ключевое правило:** `geoip` (страна → CIDR) **ставится статически** в `AllowedIPs` на коннекте — диапазоны IP не крутятся. **Живой DNS-путь зарезервирован за `geosite` / доменами**, и это вдвое снижает сложность и churn маршрутов в рантайме.

## Возможности

### Гео-раздельное туннелирование
- Маршрутизация по **geosite** (категории доменов), **geoip** (страны), отдельным **доменам**, **CIDR** или **приложению**.
- **Живой перехват DNS** переводит совпавший домен на туннель в момент его резолва — никаких устаревших снимков IP, никаких утечек CDN.
- **Переиспользуемые списки маршрутизации** — именованные наборы правил, общие для нескольких конфигов, каждый со своими исключениями, тумблером all-UDP, opt-in IPv6 и режимом.

### Два режима туннеля
- **Весь трафик** — всё через VPN, с **байпасом** RFC1918 / локальных подсетей, пользовательскими исключениями и **kill-switch** (Windows Filtering Platform).
- **Выборочно** — по умолчанию в туннель не идёт ничего; туннель берут только совпавшие гео-правила. Домены заходят вживую по мере резолва.
- **Эндпоинт всегда исключён** из собственного туннеля, чтобы несущая не замкнулась сама на себя.

### Устойчивость к DPI и надёжный коннект
- Обфусцированный транспорт **AmneziaWG** поверх WireGuard **UAPI**.
- **WebSocket-транспорт** (wstunnel поверх TCP/443) для сетей, где режут голый UDP WireGuard.
- **Дружелюбие к DDNS**: эндпоинт резолвится до туннеля, **пиннится** и откатывается к **last-known-good** IP — коннект переживает захват DNS и флапы.

### Split-horizon DNS
- Локальные имена резолвятся на **локальном** резолвере, совпавшие — на **туннельном**, остальные — **напрямую**; трёхсторонний выбор на каждый запрос.
- **IPv6** по умолчанию отключён (нет v6-утечки/блэкхола) и включается **per-список** для dual-stack.

### Конфиги и UX
- Импорт WireGuard / AmneziaWG **`.conf`**, **QR-кода** или **бандла** AmneziaGeo; экспорт бандлов для передачи.
- UI на **Avalonia** со светлой / тёмной темами и **живым переключением языка** (RU / EN).
- Встроенные **диагностика и просмотр логов**.
- **Hot-apply**: правки доменов / geoip применяются вживую без переподключения; баннер просит реконнект только когда изменение действительно этого требует.

### Эксплуатация
- **Двухпроцессная архитектура** — долгоживущий привилегированный агент-сервис плюс временный per-tunnel сервис; UI работает без прав через IPC.
- **Подписанный MSI-установщик** (WiX Burn bundle) с опциональными ярлыками на столе / в меню Пуск, **сохранением параметров установки** и **встроенным автообновлением**.

## Как это устроено

### Три уровня
1. **Управляющий уровень — C#.** Парсит `geosite.dat` / `geoip.dat`, разворачивает токены правил (`geosite:x` → доменные суффиксы, `geoip:cc` → CIDR), генерирует конфиг и запускает DNS-прокси + программатор маршрутов.
2. **Плоскость данных — Go.** Обфускационный движок AmneziaWG (`amneziawg-go`), поставляется как **непрозрачный нативный артефакт под ОС** (`tunnel.dll` на Windows). Никогда не переписывается; управляется через WireGuard UAPI (`set` / `replace_allowed_ips`).
3. **Привилегированное исполнение.** Служба Windows держит движок и программирует таблицу маршрутов ОС и файрвол; UI остаётся без прав.

### Решение о маршруте, по уровням
Трафик «отваливается» от туннеля на трёх независимых уровнях:

| Уровень | Весь трафик | Выборочно |
|---|---|---|
| **AllowedIPs** (что принимает адаптер) | `0.0.0.0/0` | только CIDR гео-правил + /32 резолверов |
| **Маршруты ОС** (исключения к физ. шлюзу) | эндпоинт + RFC1918 + локальные подсети + пользовательские | только эндпоинт |
| **DNS-прокси** (split-horizon) | LAN / туннель / напрямую | LAN / туннель / напрямую + живой роут на совпадении |
| **Kill-switch** (WFP) | вкл | выкл |

Коротко: «весь трафик» — это *туннель по умолчанию, дырки наружу*; «выборочно» — *напрямую по умолчанию, выбранное заводим внутрь* — и домены заводятся динамически, по мере того как DNS-прокси видит их резолв.

## Начало работы (Windows)

**Требования:** Windows 10 / 11 (x64). Установщик ставит службу на всю машину и требует прав администратора.

1. Скачайте свежий подписанный установщик из **[Releases](../../releases)** (`AmneziaGeo-<версия>-win-x64-*.exe`).
2. Запустите — он ставит агент-службу AmneziaGeo и приложение.
3. Запустите AmneziaGeo, **импортируйте конфиг** (файл `.conf`, QR-код или переданный бандл).
4. Выберите **список маршрутизации** (или весь трафик) и подключайтесь.

Приложение обновляет себя само через встроенный апдейтер.

## Помимо гео: приложения, UDP и прокси-фолбэк

Гео-правила — лишь один вид записей. Тот же список маршрутизации ведёт и **приложения**, и **сырой UDP**, а когда сеть режет UDP целиком — весь туннель уносит **WebSocket-прокси** поверх TCP 443.

### Маршрутизация по приложениям

Добавьте в список маршрутизации **приложение** — и в туннель пойдёт трафик *только этого приложения*, независимо от того, на какие домены и IP оно ходит. Полезно, когда нужно завернуть одну программу (профиль браузера, игру, мессенджер), а всё остальное оставить напрямую, либо когда приложение ходит на адреса, которых нет ни в одном списке `geosite`.

Приложение выбирается из выпадающего списка **Источник** в редакторе списка маршрутизации:
- **Запущенные** — выбрать из живых процессов (и служб);
- **Установленные** — выбрать из установленных программ;
- **Папка** — совпадение по всем исполняемым файлам в папке (`app:dir=…`);
- **Файл** — совпадение по одному исполняемому файлу (`app:path=…`).

Каждый выбор превращается в правило `app:` (`app:dir=` / `app:path=` / `app:svc=`). Агент следит за процессом-владельцем **и его дочерними процессами** (например, за отдельным сетевым процессом браузера) и **пиннит IP, к которым это приложение подключается**, в туннель вживую — тем же механизмом, каким DNS-прокси пиннит домены. Слишком широкие матчеры (`svchost.exe`, `C:\Windows\…`, корень диска) отклоняются, чтобы правило случайно не завернуло всю систему.

### Весь UDP через туннель

Трафик реального времени — **голосовые / видеозвонки, онлайн-игры** — обычно узнаёт IP своих серверов из in-band-сигналинга, а не из DNS, поэтому доменное правило их не видит и медиа утекает напрямую. Пер-список тумблер **Весь UDP через туннель** заворачивает *каждую* исходящую UDP-датаграмму в туннель, пока активна выборочная маршрутизация. Локальная сеть и сама несущая туннеля исключены, так что через VPN идёт только медиа.

### WebSocket-прокси — когда UDP заблокирован

AmneziaWG — это UDP. В сетях, где UDP режут или душат (много корпоративного / captive Wi-Fi, часть мобильных операторов), хендшейк не проходит. Тогда AmneziaGeo может унести весь туннель **в WebSocket поверх TCP 443** (через `wstunnel`) — это выглядит как обычный HTTPS и проходит там, где UDP не может.

Включается пер-конфиг в **Конфиг → Транспорт → WebSocket-прокси → Использовать прокси**:
- **Хост сервера** — фронт прокси (например, DNS-имя вашего эндпоинта);
- **Порт** — обычно **443**;
- **Авторизация** — **Нет**, **Basic** (логин / пароль) или **Токен**;
- **MTU** — опционально; пусто = дефолт 1280.

Применяется на следующем коннекте. Когда прокси поднят, UDP WireGuard едет внутри TCP/443-потока — рабочий туннель получают даже UDP-враждебные сети, а вместе с **Весь UDP** — и рабочий голос.

> Текущая WebSocket-несущая — единственный TCP-поток, поэтому объёмный и realtime-трафик делят одну очередь (head-of-line blocking). Транспорт **класса xray** без этой проблемы — в [роадмапе](#сервер-и-прокси).

### Собираем вместе — рабочий пример

Задача: завернуть **Discord** (вместе с голосом) в VPN на сети, которая режет UDP, а всё остальное оставить напрямую.

1. **Новый список** → назвать `discord`.
2. **Маршрутизация по приложениям** → Источник **Запущенные** → выбрать **Discord** → **Добавить** (добавит `app:dir=…\Discord`).
3. Включить **Весь UDP через туннель** — голосовые серверы Discord приходят через сигналинг, не через DNS.
4. Сеть режет UDP, поэтому открыть **Конфиг → Транспорт**, включить **Использовать прокси**, указать хост = ваш эндпоинт, порт = **443**, авторизацию как требует ваш сервер.
5. Выбрать список `discord` для подключения и подключиться.

Итог: Discord (текст + голос) идёт в туннель поверх TCP 443; всё остальное — напрямую по физическому каналу.

Общий код — **по языкам**: [`amneziageo-desktop/core`](amneziageo-desktop/core/) это .NET-библиотека, подключаемая головами Windows / Linux / Android через `<ProjectReference>` (не сабмодуль); у каждой головы своё решение. `AmneziaGeoKit` (Swift) общий для двух Apple-платформ. Git-сабмодули зарезервированы за апстрим-движками (`amneziawg-windows`, `amneziawg-apple`).

> Честная оговорка про область: «одна кодовая база на C# для всего» не переживает контакта с платформенными VPN-API. **Apple (macOS + iOS) — единый нативный Swift-проект**: NetworkExtension только нативный. C# держит Windows, Linux, Android и весь десктопный UI / оркестрацию. Два языковых мира делят **концепции** (конфиг, гео-логику), а не код.

## Роадмап

### Клиенты по платформам
Windows уже есть; остальные клиенты запланированы:

- **Windows** — ✅ текущий фокус; функционально полная бета (см. список возможностей выше).
- **Android** — 📋 план — C# (.NET) `VpnService`, движок как gomobile `.aar`.
- **Linux** — 📋 план — UI на C# (Avalonia) + userspace-демон `amneziawg-go` через UAPI, управляемый небольшим привилегированным хелпером (systemd / `setcap`); гео-сплит, kill-switch через `nftables`. В основном out-of-process оркестрация, без P/Invoke.
- **macOS** — 📋 план — нативный Swift поверх нативного NetworkExtension, общий кор `AmneziaGeoKit`.
- **iOS** — 📋 план — тот же нативный кор (`AmneziaGeoKit`), один Xcode-проект вместе с macOS.

### Сервер и прокси
Планируется самохостируемый **серверный сервис**, чтобы AmneziaGeo был не только клиентом, — он провижинит и обслуживает туннели, к которым подключаются клиенты:
- **Провижининг AmneziaWG-сервера** — поднять сервер и раздавать конфиги.
- **Пер-девайс пиры** — отдельный ключ / адрес / слот пира на каждое устройство, чтобы два устройства не дрались за один эндпоинт.
- **Прокси-транспорт** — WebSocket и транспорт **класса xray** для сильной устойчивости к DPI и чистого **UDP / голоса** (без head-of-line blocking единственной TCP-несущей).
- **Раздача конфигов** — провижининг через бандл / QR для онбординга.

### Клиент
- Более широкая маршрутизация по приложениям и app-aware правила.
- Умная маршрутизация загрузок (объёмные передачи мимо туннеля, когда это выгодно).
- Больше вариантов транспорта в UI.

Требования: **.NET 10 SDK**, Windows и WiX toolset для установщика.

```powershell
# UI + агент
dotnet build amneziageo-desktop/windows/AmneziaGeo.Windows.Ui/AmneziaGeo.Windows.Ui.csproj -c Release
```

## Приватность

AmneziaGeo не собирает аналитику и телеметрию и не отправляет авторам никаких персональных данных. Программа устанавливает исходящие соединения только с:

- **VPN-эндпоинтом(ами)** из импортированного вами конфига;
- **сервером обновлений**, на который она настроена;
- **источниками гео-баз**, которые вы настроили, — чтобы скачать данные `geosite` / `geoip`.

Всё остальное маршрутизируется строго по вашим правилам. Данные об использовании не покидают машину.
